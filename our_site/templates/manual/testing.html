¿Qué es el testing?

Conjunto de actividades para verificar que el software hace lo que debe hacer y validar que satisface las necesidades del usuario, antes y después de desplegarlo.

Detectar defectos temprano
Prevenir regresiones
Aumentar confianza
Documentación viva

Tipos de pruebas (con definición)

  • Pruebas unitarias: validan un componente pequeño (función, clase o método) de forma aislada. Son rápidas y dan feedback inmediato.
  • Pruebas de integración: verifican que múltiples módulos funcionan correctamente cuando se combinan (por ejemplo, servicio + base de datos).
  • Pruebas funcionales (o end‑to‑end): evalúan el sistema desde la perspectiva del usuario, comprobando flujos completos (login, compra, etc.).
  • Pruebas automatizadas: casos de prueba que se ejecutan por herramientas sin intervención humana (por ejemplo, en CI/CD). Pueden ser unitarias, de integración o funcionales, pero ejecutadas automáticamente.

Nota: también existen otras como regresión, rendimiento, seguridad, según las necesidades del proyecto.

Importancia del testing en proyectos

  • Reduce costos: arreglar errores temprano es mucho más barato que en producción.
  • Permite refactorizar con seguridad: una buena base de pruebas detecta roturas al cambiar el código.
  • Mejora la calidad: ayuda a cubrir casos borde y requisitos críticos.
  • Aumenta la confianza del equipo y del cliente: despliegues más predecibles y estables.
  • Documentación ejecutable: las pruebas explican cómo debe comportarse el sistema.

Buenas prácticas

  • Escribir pruebas antes o junto al código (TDD o enfoque simultáneo): define claramente el comportamiento esperado y guía el diseño.
  • Patrón AAA (Arrange‑Act‑Assert): prepara datos, ejecuta la acción y verifica el resultado para pruebas legibles.
  • Pruebas pequeñas, independientes y deterministas: que no dependan del orden de ejecución ni de internet o el reloj del sistema.
  • Usar dobles de prueba (mocks, stubs, fakes): aísla dependencias externas como BD, APIs o reloj.
  • Nombrar claramente: un buen nombre describe qué se prueba y qué se espera.
  • Medir cobertura con criterio: alta cobertura ayuda, pero la meta real es validar comportamientos críticos.
  • Integrar en CI/CD: ejecutar la suite automáticamente en cada commit y al desplegar.

Mini‑ejemplo (pseudo‑código)

Ilustración de una prueba unitaria usando el patrón AAA:

// Arrange
const carrito = new Carrito();
carrito.agregar({ id: 1, precio: 10 });

// Act
const total = carrito.total();

// Assert
assert.equal(total, 10);

El ejemplo muestra la idea general; usa tu framework favorito (p.ej., JUnit, pytest, PHPUnit, Jest).
